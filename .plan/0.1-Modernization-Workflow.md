# BitchX Modernization — Agent Workflow Instructions

> **Author:** Mark LaPointe <mark@cloudbsd.org>
>
> **Sponsorship:** None

> **Purpose:** This document defines how autonomous agents (and human developers) should consume the plan documents, pick tasks, claim work, commit changes, and handle merge conflicts when multiple agents work on the same branch concurrently.
>
> **Note for agents:** Read this document **before** starting any task. Follow these instructions exactly. If any step is unclear, ask for clarification.

---

## 0.1.1 Overview

This project uses a **shared branch model** where multiple agents may be working on the same Git branch simultaneously. To avoid conflicts and ensure traceability, every task must be claimed, tracked, and completed following a strict protocol.

**Key principles:**
- **Claim before work** — Never start a task without first claiming it in the document
- **Check dependencies** — Never start a task whose dependencies are not yet complete
- **Commit frequently** — Commit after each task completion, not after every file edit
- **Handle conflicts gracefully** — If a merge conflict occurs, determine if the task was taken by another agent

## 0.1.2 Task Selection

### 0.1.2.1 How to Pick a Task

1. **Consult the TOC** — Open [`00-MODERNIZATION-TOC.md`](./00-MODERNIZATION-TOC.md) to understand the document structure
2. **Read the relevant document** — Open the document containing the task you want to work on
3. **Find an unclaimed task** — Look in the task tables for a row where:
   - `Status` is empty or `⬜ PENDING`
   - `Assigned To` is empty
   - `Start` is empty
4. **Check dependencies** — Verify that all tasks listed in the `Dependencies` column are marked as `✅ DONE` or have a completed `End` date
5. **If dependencies are not met** — Do NOT start the task. Either:
   - Pick a different task whose dependencies are met, OR
   - Work on completing the blocking dependency first

### 0.1.2.2 Priority Order

Tasks should generally be worked on in the following order:

1. **Phase P0 tasks** (core infrastructure, security fixes) — highest priority, blocks everything
2. **Phase P1 tasks** (protocol implementation) — blocks UI and scripting
3. **Phase P2 tasks** (DCC and file transfers) — blocks advanced features
4. **Phase P3 tasks** (UI and scripting) — user-facing features
5. **Phase P4-P5 tasks** (database, IPC) — backend systems
6. **Phase P6-P10 tasks** (testing, deployment, documentation) — quality assurance

Within each phase, tasks with the most dependencies should be started first.

## 0.1.3 Claiming a Task

### 0.1.3.1 Before You Start

Before claiming a task, ensure:

- [ ] You have read this workflow document
- [ ] You have read the relevant plan document(s)
- [ ] You understand the task requirements
- [ ] All dependencies are marked as complete
- [ ] No other agent has claimed the task (check `Assigned To` column)

### 0.1.3.2 How to Claim

To claim a task, update the task table row in the document:

1. **Set `Status`** to `🔄 IN PROGRESS`
2. **Set `Assigned To`** to your agent identifier using the system hostname (e.g., `freedev002`). Use the `hostname` command to identify your system.
3. **Set `Start`** to the current date/time in ISO 8601 format: `YYYY-MM-DD HH:MM UTC`
4. **Leave `End` empty** — will be filled on completion

**Example — Before claiming:**

| ID | Task | Status | Assigned To | Owner | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-------|-----|--------------|-------|-------|
| P0.1 | Set up Rust project with Cargo | ⬜ PENDING | | | | | | `Cargo.toml`, `src/main.rs` | |

**Example — After claiming:**

| ID | Task | Status | Assigned To | Owner | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-------|-----|--------------|-------|-------|
| P0.1 | Set up Rust project with Cargo | 🔄 IN PROGRESS | freedev002 | | 2026-04-30 22:00 UTC | | | `Cargo.toml`, `src/main.rs` | |

### 0.1.3.3 Commit and Push After Claiming

After claiming a task, immediately commit and push the change:

```bash
git add .plan/<document>.md
git commit -m "Claim task <ID>: <brief description>"
git push
```

**Important:** You must push immediately after committing. Other agents working on the same branch need to see your claim to avoid starting the same task. The claim is not effective until it is pushed to the remote repository.

## 0.1.4 Working on a Task

### 0.1.4.1 Implementation Guidelines

1. **Follow the plan** — Implement exactly what the task describes. Do not add scope.
2. **Follow code style** — Match the Rust idioms defined in [`03-Rust-Architecture.md`](./03-Rust-Architecture.md)
3. **Write tests** — Every task that involves code changes must include tests
4. **Update cross-references** — If you change a section number or file name, update all cross-references
5. **Document as you go** — Update documentation/comments in code, not just plan documents

### 0.1.4.2 If You Get Stuck

If you encounter an issue that blocks progress:

1. **Try 3 genuine fix attempts** — Document what you tried and why it didn't work
2. **Mark the task as blocked** — Set `Status` to `🟡 BLOCKED` and add the reason to `Notes`
3. **Commit and push** — So other agents know the task is blocked
4. **Ask for guidance** — Use the `ask_user_v2` tool to explain the issue

### 0.1.4.3 If the Task is Impossible

If a task cannot be completed as described (e.g., design flaw, missing dependency, technical limitation):

1. **Mark the task as `❌ FAILED`** — Set `Status` to `❌ FAILED`
2. **Document why** — Add the reason to `Notes`
3. **Commit and push**
4. **Move to the next task** — Do not block on failed tasks unless they are critical dependencies

## 0.1.5 Completing a Task

### 0.1.5.1 How to Mark Complete

When a task is fully implemented and verified:

1. **Run all unit tests** — Execute the full test suite for the affected module(s). Do NOT consider a task complete unless **all** unit tests pass.
2. **Fix other agents' code if needed** — If tests fail due to issues in another agent's code (e.g., a dependency task has bugs), fix those issues as part of completing your task. Do not leave broken tests for someone else.
3. **Set `Status`** to `✅ DONE`
4. **Set `End`** to the current date/time in ISO 8601 format: `YYYY-MM-DD HH:MM UTC`
5. **Update `Notes`** — Add a brief summary of what was implemented (1-2 sentences), including any fixes made to other agents' code
6. **Update `Files`** — Ensure all files created/modified are listed, including files modified in other agents' tasks

**Example — After completion:**

| ID | Task | Status | Assigned To | Owner | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-------|-----|--------------|-------|-------|
| P0.1 | Set up Rust project with Cargo | ✅ DONE | freedev003 | | 2026-04-30 22:00 UTC | 2026-04-30 22:15 UTC | | `Cargo.toml`, `src/main.rs` | Created minimal Cargo project with Tokio runtime |

### 0.1.5.2 Commit and Push After Completion

After marking a task complete, commit and push:

```bash
git add .
git commit -m "Complete task <ID>: <brief description>"
git push
```

## 0.1.6 Handling Merge Conflicts

### 0.1.6.1 If You Receive a Merge Conflict

If you pull and get a merge conflict:

1. **Check the task table** — Determine which task the conflicting changes relate to
2. **Identify the winner** — If another agent completed a task you were also working on, their changes win
3. **Reconcile** — If the conflict is about task status (not code), keep the more complete status (e.g., `✅ DONE` over `🔄 IN PROGRESS`)
4. **Commit the resolution** — After resolving, commit and push

### 0.1.6.2 If Another Agent Claims Your Task

If you pull and see that another agent has claimed a task you were about to start:

1. **Do not overwrite their claim** — Find a different task
2. **Check for dependencies** — Perhaps completing that task will unblock others
3. **Move on** — Do not block waiting for them

## 0.1.7 Communication

### 0.1.7.1 Status Updates

After completing significant milestones (not every task), post a status update:

- What you completed
- What you're working on next
- Any blockers or concerns

### 0.1.7.2 Blocking Issues

If a task is blocked by external factors (waiting on library support, waiting on API changes, etc.):

1. Mark the task as `🟡 BLOCKED`
2. Document the blocking issue in `Notes`
3. Move to a different task if possible

## 0.1.8 Code Quality Standards

### 0.1.8.1 Rust Code Standards

- Use `cargo clippy` to check for common mistakes
- Use `cargo fmt` to format code consistently
- Document public APIs with doc comments
- Write unit tests for all public functions
- Use proper error handling with `Result` and `?` operator

### 0.1.8.2 Security Standards

- **No unsafe code** unless absolutely necessary and documented
- All external input must be validated
- Use Rust's type system to enforce invariants
- Follow the security guidelines in [`02-Security-Audit.md`](./02-Security-Audit.md)

## 0.1.9 Commit Message Format

Use this format for commit messages:

```
<type>: <brief description>

<optional detailed description>

Closes #<issue-number>
```

**Types:**
- `feat:` New feature
- `fix:` Bug fix
- `refactor:` Code refactoring
- `docs:` Documentation changes
- `test:` Test changes
- `chore:` Build system or tooling

**Example:**

```
feat: Implement SASL PLAIN authentication

Implements SASL PLAIN mechanism for IRCv3 authentication.
Handles base64 encoding of credentials and server challenges.

Closes #42
```

## 0.1.10 Rollback Procedure

### 0.1.10.1 If a Task Introduces Bugs

If your completed task introduces bugs that cannot be fixed quickly:

1. **Revert the commit** — `git revert <commit-hash>`
2. **Mark the task as `🟡 BLOCKED`**
3. **Investigate** — Understand the root cause
4. **Fix and retry** — Implement a corrected solution

## 0.1.11 Final Checklist

Before marking a task complete, verify:

- [ ] Code compiles without errors
- [ ] All unit tests pass
- [ ] Code is formatted (`cargo fmt`)
- [ ] No clippy warnings
- [ ] Documentation updated if API changed
- [ ] Task table updated with status, dates, and notes
- [ ] Commit pushed to remote
